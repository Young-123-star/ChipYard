## 1. 背景与目标

第一阶段三大块「资源管理 + 入住/退宿 + 费用」中，资源管理与入住已完成。本 spec 覆盖「退宿」。

退宿的数据来源对接 OA，两条入口：
- **OA 退宿申请单**：员工主动申请退宿，审批通过后推送给本系统。
- **OA 离职单**：员工离职，需腾退宿舍并同步人员离职状态。

与入住对称，采用**防腐层 + 预留接口**：外部报文经适配器映射成内部模型，内部退宿逻辑只认本系统标准模型。

费用结算（水电/赔偿）依赖尚未建设的费用模块，**不在本期范围**；本期退宿主链路为：退宿单 → 管理员办理退宿（释放床位 + 归档在住档案）。

**成功标准：**
- 后端 `mvn spring-boot:run` 仍零外部依赖启动（H2 + 种子）。
- 可向两个 webhook 投样例报文，按匹配级联生成"待退宿单"或返回明确结果（无居住记录 / 无在住记录 / 离职无在住）。
- 管理员可手工新建退宿单；可办理退宿，床位释放（含 `current_user_id` 置空持久化）、房间统计刷新、在住档案置已退宿并回填退宿日、退宿单置已退宿，看板/床位页自动反映。
- 重复推送同一业务单据幂等。
- 后端关键逻辑有测试覆盖。

## 2. 范围与非目标

**本子项目做：**
- 退宿单 `dms_checkout_order`：OA 退宿申请/OA 离职/手工三来源，待退宿载体。
- OA 退宿申请单 + OA 离职单 webhook + 适配器（防腐层）+ 幂等去重 + 匹配级联。
- 离职单同步将居住人置为离职（`dms_resident.status=0`）。
- 办理退宿事务：释放床位 + 刷新房间统计 + 在住档案置已退宿 + 回填退宿日 + 退宿单置已退宿。
- 修复 `Bed.currentUserId` 的 null 更新策略（入住子项目遗留占位）。
- 前端「入住管理」菜单组下新增「退宿单」页。

**非目标（留待后续）：**
- 费用结算（水电/物业/赔偿）—— 依赖费用子项目。
- 退宿检查（家具完好/水电读数/钥匙归还/损坏赔偿登记）—— 后续增强或并入费用子项目。
- 多节点审批流、员工自助端、OA 真实对接实现（报文映射细节、真实鉴权、出站回写）—— 本期仅预留。
- 集成事件表/重试/审计。

## 3. 架构与模块结构

沿用模块化单体 + package-by-feature，对称 `checkin`。

```
module/
├── checkout/          # 退宿核心：退宿单(order) + 办理退宿
│   └── controller/service/mapper/entity/dto/vo
└── integration/oa/    # 扩充：退宿申请/离职两个 webhook + 适配器 + 外部 DTO
```

**模块边界：**
- `integration` 适配器只调用 `checkout`/`resident` 的 Service，不碰 mapper。
- `checkout` 办理退宿通过 `resource` 模块的 `BedService.release` / `RoomService.refreshOccupancy` 操作床位与房间，不直接写表；通过 `checkin` 模块读取/更新在住档案（经 CheckinService 接口）。

> `checkout` 需要读取并更新在住档案（`dms_checkin_record`）。为不跨模块直接访问 mapper，在 `CheckinService` 上新增两个方法：`findActiveRecordByResident(residentId)`（取居住人当前在住档案）与 `markCheckedOut(recordId, checkoutDate)`（置已退宿+回填退宿日）。

## 4. 数据模型

通用字段：created_at/updated_at/deleted_at（逻辑删除），状态用 TINYINT。

### 4.1 dms_checkout_order 退宿单（新增）
| 字段 | 类型 | 说明 |
|---|---|---|
| id | BIGINT PK | |
| biz_no | VARCHAR(64) 唯一 | OA 退宿申请单号/离职单号；手工 `MANUAL-...`，**幂等键** |
| resident_id | BIGINT | 居住人 |
| checkin_record_id | BIGINT | 要腾退的在住档案（离职无在住时可空，但有 record 才能办理）|
| source | TINYINT | 1退宿申请 2离职 3手工 |
| reason | VARCHAR(500) | 原因 |
| expect_checkout_date | DATE | 预计退宿日 |
| status | TINYINT | 1待退宿 2已退宿 3已取消 |
| raw_payload | CLOB | 原始报文 |
| 通用字段 | | |

### 4.2 对现有表/实体的改动
- `dms_checkin_record` 新增列 `checkout_date DATE`（办理退宿时回填），实体 `CheckinRecord` 加 `LocalDate checkoutDate`。
- `Bed.currentUserId` 加 `@TableField(value = "current_user_id", updateStrategy = FieldStrategy.ALWAYS)`，使 `release()` 的 `current_user_id=null` 能持久化。
- 离职处理：`dms_resident.status` 置 0（离职）。

## 5. 对接边界（防腐层）

### 5.1 Webhook 端点
```
POST /api/integration/oa/checkout-application   # OA 退宿申请单
POST /api/integration/oa/resignation            # OA 离职单
```
- 鉴权：复用 `X-Integration-Token`（与入住同一 `IntegrationTokenGuard`）。
- 各自独立外部 DTO，由 `CheckoutApplicationAdapter` / `ResignationAdapter` 映射处理。
- 幂等：`biz_no` 唯一 + 入库前查重；重复推送返回已存在退宿单（幂等）。
- 原始报文存入 `order.raw_payload`。

### 5.2 匹配级联与返回结果
适配器返回 `CheckoutResultVO { outcome, orderId, residentResigned, message }`，`outcome` 取值：

| 情况 | 退宿申请单 | 离职单 |
|---|---|---|
| 工号无对应居住人 | `NO_RESIDENT`（message「无居住记录」）| `NO_RESIDENT`（「无居住记录」）|
| 有居住人、无在住记录 | `NO_ACTIVE_CHECKIN`（「无在住记录」，不建单）| 置居住人离职 + `RESIGNED_NO_CHECKIN`（不建单，residentResigned=true）|
| 有居住人、有在住记录 | `ORDER_CREATED`（建待退宿单，关联在住档案）| `ORDER_CREATED` + 置居住人离职（residentResigned=true）|

> 在住记录查找：`CheckinService.findActiveRecordByResident(residentId)` 返回该居住人 status=1（在住）的档案；无则视为"无在住记录"。webhook HTTP 状态均为 200，结果体现在 `CheckoutResultVO.outcome`（便于 OA 侧据此处理）。

### 5.3 出站预留
复用入住子项目的 `OaClient`（可加 `notifyCheckoutCompleted(bizNo)` 空实现），办理退宿后调用；将来填真实回写。

## 6. 退宿状态流转

```
[Webhook(退宿申请/离职) / 手工] ──> 退宿单(1 待退宿)
                                       │ 管理员办理退宿（@Transactional）
                                       │   校验：退宿单待退宿 + 关联在住档案存在且在住
                                       │   → 释放床位(空闲 + current_user_id=null)
                                       │   → 刷新房间统计(occupied_beds/status)
                                       │   → 在住档案置已退宿(2) + 回填 checkout_date
                                       │   → 退宿单置已退宿(2)
                                       ▼
                                 退宿单(2 已退宿) / 档案(2 已退宿) / 床位空闲
   退宿单(1 待退宿) ──管理员取消──> 退宿单(3 已取消)
```

办理退宿为单事务，任一步失败整体回滚。

## 7. API 设计

路径前缀 `/api`，统一 `R<T>`。webhook 走 `X-Integration-Token`，其余需 Bearer JWT。

| 方法 | 路径 | 说明 |
|---|---|---|
| POST | /api/integration/oa/checkout-application | OA 退宿申请单 webhook（返回 R<CheckoutResultVO>）|
| POST | /api/integration/oa/resignation | OA 离职单 webhook（返回 R<CheckoutResultVO>）|
| GET | /api/checkout/orders | 退宿单分页列表（筛选：状态/来源）|
| POST | /api/checkout/orders | 手工新建退宿单（传 residentId；后端解析其在住档案，无在住则拒绝）|
| POST | /api/checkout/orders/{id}/confirm | 办理退宿（核心事务）|
| POST | /api/checkout/orders/{id}/cancel | 取消退宿单 |

错误处理沿用：BizException → 业务码 + 提示；webhook token 错误 → 401（code 40100）。

## 8. 前端设计

「入住管理」菜单组下新增「退宿单」页（沿用蓝主题 + ProTable 模式）：
- 列表：业务号 / 居住人（姓名+工号）/ 来源 / 原因 / 预计退宿 / 状态（待退宿·已退宿·已取消）。
- `待退宿` 行操作：**办理退宿**（确认弹窗显示居住人 + 所退房间/床位信息）、**取消**。
- **手工新建**弹窗：选居住人（仅在住的）→ 提交（后端解析其在住档案）。
- 办理退宿后，看板页/床位页对应床位自动恢复空闲（同一套 bed/room 数据，无需额外改动）。
