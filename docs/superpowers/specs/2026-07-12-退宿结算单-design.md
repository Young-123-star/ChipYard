# 退宿结算单首版 — 设计方案

> 日期：2026-07-12
> 状态：已确认，待实施

## 1. 目标与范围

在现有“退宿欠费挂账”基础上增加正式退宿结算单，统一处理：

- 历史未缴住宿费、水费和电费；
- 退宿当月住宿费自然日折算；
- 人工补收与人工减免；
- 现场结清或欠费挂账；
- 挂账后的整单缴费结清；
- 结算确认人、后续结清人及操作时间记录。

结算模块集中负责计算、快照、账单关联、状态流转和退宿确认，避免规则散落在 Controller、费用模块或前端。

业务状态流：

```text
待退宿 → 结算草稿
              ├─ 现场结清 → 已结清 → 完成退宿
              └─ 欠费挂账 → 已挂账 → 后续整单缴费 → 已结清
```

## 2. 核心规则

### 2.1 结算范围

- 汇总该入住档案下状态为未缴的住宿费、电费、水费及其他已生成费用。
- 已缴和已作废账单不进入本次结算。
- 水电费首版只纳入已经生成的个人账单，不在退宿时自动进行房间末次抄表。
- 人工调整分为补收和减免，金额均以正数保存，通过方向区分。
- 人工减免不得超过本次应收，首版不产生退款。

### 2.2 当月住宿费折算

计算公式：

```text
月住宿费 ÷ 当月自然日数 × 当月实际居住天数
```

- 入住日和退宿日均计费，金额使用 `HALF_UP` 保留两位小数。
- 入住日在退宿月份之前时，从当月 1 日计算；同月入住时，从入住日计算。
- 当月没有住宿费账单时，确认结算时创建折算账单。
- 当月存在未缴整月住宿费账单时，确认结算时作废原账单并创建折算账单。
- 当月住宿费已经缴清时保持原账单，不重新折算、不退款。
- 缺少对应房型收费标准时禁止生成结算草稿。

### 2.3 结算结果

- 现场结清：应付金额大于零时必须选择现金或转账；应付为零时不要求支付方式。
- 欠费挂账：结算内账单统一转为挂账，退宿仍正常完成并释放床位。
- 已挂账结算只能整单缴费，不支持部分缴费。
- 结算确认后快照不可编辑；冲销、反结算和退款不在首版范围。

## 3. 数据模型

### 3.1 结算主表 `dms_checkout_settlement`

保存：

- 结算单号；
- 退宿单、入住档案和居住人；
- 实际退宿日期；
- 应收金额、减免金额、应付金额、实收金额和挂账金额；
- 处理方式和支付方式；
- 状态：1 草稿、2 已结清、3 已挂账、4 已取消；
- 确认人、确认时间、结清人和结清时间；
- 备注及通用审计字段。

约束：

- 结算单号唯一；
- 同一退宿单只能存在一张有效结算单。

### 3.2 结算明细表 `dms_checkout_settlement_line`

保存：

- 所属结算单；
- 明细类型：历史账单、折算住宿费、人工补收、人工减免；
- 方向：补收或减免；
- 项目名称、所属账期、金额和说明；
- 来源账单 ID、账单号、账单类型和金额快照。

同一来源账单在一张结算单中只能出现一次。

### 3.3 账单扩展

- `dms_fee_bill` 增加 `settlement_id`，标识账单已进入退宿结算。
- 账单类型增加“退宿调整”。
- 人工补收减去人工减免后生成一张净调整账单；具体组成仍保存在结算明细中。
- 结算内账单禁止单张缴费或作废，只能通过整张结算单处理。
- 净调整账单允许负金额，使现有应收、未缴和收缴率报表继续按净额统计。

数据库使用 V7 Flyway 迁移，并同步 H2 `schema.sql` 和演示数据。

## 4. 模块接口与事务

退宿结算作为 `checkout` 下的独立模块，对外提供小接口，内部统一协调退宿、费用、入住档案和房源状态。

### 4.1 对外接口

- `PUT /api/checkout/orders/{orderId}/settlement`
  - 新建或重新计算草稿；
  - 请求包含实际退宿日期和人工调整数组；
  - 只有草稿状态允许重复保存。
- `GET /api/checkout/orders/{orderId}/settlement`
  - 返回结算主数据和结构化明细。
- `POST /api/checkout/orders/{orderId}/settlement/confirm`
  - 请求包含处理方式；
  - 现场结清且应付大于零时必须提供支付方式。
- `POST /api/checkout/settlements/{settlementId}/pay`
  - 将已挂账结算整单缴费并同步更新关联账单。

现有 `/api/checkout/orders/{id}/confirm` 保留一个版本用于兼容：仅传退宿日期时，自动生成无人工调整的草稿并按“欠费挂账”确认。新前端只使用结算接口。

### 4.2 草稿计算

保存草稿时：

1. 校验退宿单处于待退宿状态，入住档案仍在住。
2. 校验退宿日期不早于入住日期且不晚于当天。
3. 读取历史未缴账单并保存金额快照。
4. 根据收费标准计算当月折算住宿费。
5. 校验人工调整的类别、说明和金额。
6. 计算应收、减免和最终应付。
7. 替换原草稿明细，不修改正式账单和房态。

### 4.3 确认结算事务

确认结算必须在同一事务内完成：

1. 重新校验退宿单、入住档案和草稿状态。
2. 校验草稿引用账单的状态和金额未发生变化；变化时要求重新计算。
3. 作废需要替换的未缴整月住宿费账单。
4. 创建折算住宿费账单和净调整账单。
5. 关联全部结算账单。
6. 根据处理方式统一置为已缴或挂账。
7. 记录确认人、确认时间、支付方式及相应金额。
8. 释放床位、刷新房态、归档入住档案。
9. 完成退宿单，并继续回填现有 `arrearsAmount` 保持列表兼容。

任一步失败时全部回滚。重复确认返回已有结果，不重复收费或释放床位。

### 4.4 挂账结清

- 仅已挂账结算允许整单缴费。
- 缴费时将全部关联挂账账单同步转为已缴。
- 记录结清人、结清时间、支付方式和实收金额。
- 结算状态由已挂账变为已结清。

## 5. 前端功能

继续使用退宿管理页，不新增独立菜单。

- 列表增加结算状态、应付金额和挂账金额，并支持结算状态筛选。
- 待退宿记录显示“办理结算”。
- 已挂账记录显示“结清”。
- 已结清和已挂账记录均可查看结算快照。
- 办理结算弹窗分为退宿信息、费用明细和处理方式三个区域。
- 页面实时显示应收、减免和最终应付。
- 默认处理方式为欠费挂账，避免误将欠费标记为已收。
- 账单页对结算内账单隐藏单张缴费和作废操作，并提示到退宿结算中处理。

## 6. 测试与验收

- 折算计算：跨月入住、同月入住、大小月、闰年、起止日计费和金额舍入。
- 当月账单：无账单、未缴整月账单替换、已缴账单不重算。
- 历史费用：住宿费、水费、电费汇总，已缴和已作废排除。
- 人工调整：补收、减免、净调整为正/负/零、减免超过应收时拒绝。
- 状态流转：草稿重算、现场结清、挂账、挂账后结清、取消和非法流转。
- 并发变化：草稿生成后账单已缴、作废或金额变化时确认失败。
- 事务回滚：账单处理、床位释放或档案归档任一步失败时全部回滚。
- 账单守卫：结算内账单禁止单张缴费和作废。
- 幂等：重复保存草稿、重复确认和重复结清不产生重复数据。
- 数据库约束：结算单号、退宿单有效结算和来源账单唯一约束。
- Controller：鉴权、参数校验、结构化明细和兼容确认接口。
- 页面冒烟：现场结清、补收减免、欠费挂账、挂账结清、已缴当月住宿费不退款。
- 执行后端全量测试和前端生产构建；部署后确认 Flyway V7 成功。

## 7. 首版暂缓
