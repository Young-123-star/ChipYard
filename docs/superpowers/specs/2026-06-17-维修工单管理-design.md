# 维修工单管理 — 设计文档

> 第二阶段第七个子项目。日期 2026-06-17。分支 `feat/repair-module`（从最新 main 切出）。

## 1. 目标与范围

为 DMS 增加**维修工单管理**：管理员手工建单，记录房间报修 → 受理（指派处理人）→ 完成的工单生命周期。全新独立模块，仅管理员操作，**不联动房间状态、不纳入维修费用、不做外部对接**。仍沿用 H2 内存库 DEMO，后端 TDD。

**验收标准：**
- 管理员可新建维修工单（房间必填、报修人选填），系统生成工单号、初始为「待受理」。
- 状态流转受守卫约束：待受理→受理（指派处理人）→处理中→完成（填结果）；待受理/处理中可取消；非法流转（如已完成再受理/取消）被拒绝。
- 列表可按状态、紧急程度、房间筛选，回显房间号/楼栋名/报修人名。
- 后端关键流转逻辑测试覆盖；全量回归通过。前端 `vue-tsc --noEmit` 零错误。

## 2. 架构与模块

- 新增 `repair` 模块（package-by-feature，照 `checkout` 模板）：`entity / mapper / dto / vo / service / controller`。
- **模块隔离**：`RepairServiceImpl` 仅通过 `RoomService`、`ResidentService` 接口做存在性校验与名称回显，不跨模块直接访问 mapper。
- **新增 1 表** `dms_repair_order`，无其他模块改动（解耦：不改房间状态、不碰 fee）。
- 前端新增菜单组「维修管理」+ 一页 `views/repair/index.vue`。

## 3. 数据模型

新增表 `dms_repair_order`（继承 `BaseEntity` 的 `created_at`/`updated_at`/`deleted_at` 逻辑删除）：

| 字段 | 类型 | 说明 |
|---|---|---|
| id | BIGINT PK AUTO_INCREMENT | |
| order_no | VARCHAR(64) NOT NULL | 工单号，如 `RO-202606-001`（建单生成） |
| room_id | BIGINT NOT NULL | 报修房间（必填） |
| resident_id | BIGINT | 报修居住人（选填） |
| title | VARCHAR(100) NOT NULL | 故障简述 |
| description | VARCHAR(500) | 故障详细描述 |
| priority | TINYINT DEFAULT 1 | 紧急程度 1普通 / 2紧急 |
| status | TINYINT DEFAULT 1 | 1待受理 / 2处理中 / 3已完成 / 4已取消 |
| handler | VARCHAR(50) | 处理人（受理时填，自由文本） |
| accepted_at | DATETIME | 受理时间 |
| result | VARCHAR(500) | 处理结果（完成时填） |
| completed_at | DATETIME | 完成时间 |
| remark | VARCHAR(200) | 备注 |

- 金额无关（不记维修费）。
- 种子：2~3 条样例工单（覆盖不同状态：待受理/处理中/已完成），挂已有房间（如 room 2 = A102），便于开箱演示列表/看板。

**工单号规则**：`RO-yyyyMM-{当月序号}`，当月序号 = 当月已建工单数 + 1，零填充 3 位（DEMO 单机够用，不考虑并发竞态）。

**字典（前端 `dict.ts`）**：
- `REPAIR_STATUS`：1 待受理 / 2 处理中 / 3 已完成 / 4 已取消（带 tag 色）。
- `REPAIR_PRIORITY`：1 普通 / 2 紧急。

## 4. 接口与状态流转

均为 JWT、挂 `/api/repair`。状态流转用独立动作端点（方案 A，沿用「退宿单 confirm/cancel」「账单 pay/void」风格），各自带状态守卫，非法流转抛 `BizException`，统一 `R<T>` / `ResultCode` 响应。

| 端点 | 方法 | 行为 / 守卫 |
|---|---|---|
| `/api/repair/orders` | GET | 分页列表；筛选 `status` / `priority` / `roomId`；返回 `PageResult<RepairOrderVO>` |
| `/api/repair/orders/{id}` | GET | 工单详情 |
| `/api/repair/orders` | POST | 新建 → status=1 待受理，生成 `order_no`；校验 room 存在、resident（若填）存在 |
| `/api/repair/orders/{id}/accept` | POST | **仅 1 待受理可受理** → status=2 处理中，写 `handler`（必填）+ `accepted_at` |
| `/api/repair/orders/{id}/complete` | POST | **仅 2 处理中可完成** → status=3 已完成，写 `result` + `completed_at` |
| `/api/repair/orders/{id}/cancel` | POST | **仅 1 待受理或 2 处理中可取消** → status=4 已取消（已完成不可取消） |

**DTO/VO：**
- `RepairCreateDTO`：roomId（必填）、residentId（选填）、title（必填）、description、priority、remark。
- `AcceptDTO`：handler（必填）。
- `CompleteDTO`：result（必填）。
- `RepairQuery`：status、priority、roomId、page、size。
- `RepairOrderVO`：实体字段 + `roomNumber`（`RoomService` 回显）+ `buildingName`（`BuildingService` 回显）+ `residentName`（`ResidentService` 回显，无报修人则空）。

## 5. 前端

- 菜单：新增菜单组「维修管理」+ 页面 `views/repair/index.vue`。
- 列表：工单号 / 房间（楼栋-房间号）/ 报修人 / 标题 / 紧急程度 tag / 状态 tag / 处理人 / 受理-完成时间；顶部按状态 + 紧急程度 + 房间筛选。
- 操作按状态显隐：
  - 待受理 → 「受理」（弹窗填处理人）。
  - 处理中 → 「完成」（弹窗填处理结果）。
  - 待受理 / 处理中 → 「取消」。
  - 随时「新建工单」（楼栋→房间联动选 + 选填报修人 + 标题/描述/紧急程度/备注）。
- `api/repair.ts`（list/get/create/accept/complete/cancel）；`types.ts` 加 `RepairOrder` 接口；`dict.ts` 加 `REPAIR_STATUS` / `REPAIR_PRIORITY` + 复用 `labelOf`/`tagTypeOf`。

## 6. 测试

**后端**（TDD）：
- `RepairOrderService`：建单（生成单号格式 + status=1 + room/resident 校验）、accept（仅待受理、写 handler+accepted_at、非待受理拒绝）、complete（仅处理中、写 result+completed_at、非处理中拒绝）、cancel（待受理/处理中可取消、已完成拒绝）、列表筛选。
- `RepairController`：六接口鉴权（无 token → 401）+ 建单/受理/完成 happy path。
- 全量回归通过。
- 测试用 `@SpringBootTest @Transactional`（自动回滚），按需插入工单构造断言。

**前端**：`vue-tsc --noEmit` 零错误（纯表单/表格，无图表）。
