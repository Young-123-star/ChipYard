## 1. 背景与目标

第一阶段三大块为「宿舍资源管理 + 入住/退宿 + 费用」。资源管理 DEMO 已完成。入住/退宿/费用三者强依赖（入住产生居住关系 → 费用对居住人计费 → 退宿结算并释放床位），单个 spec 装不下，故按依赖顺序拆为三个子项目逐个推进。**本 spec 只覆盖「入住」。**

入住的数据来源不再是管理员纯手工，而是对接外部系统：
- **OA**：OA 上有「入住申请单」，审批通过后推送给本系统，据此生成入住信息。
- **HCP（HR 系统）**：新员工入职时推送人员信息，据此生成入住信息。

目前尚无 OA/HCP 的对接细节（报文格式、地址、鉴权），因此采用**防腐层 + 预留接口**：内部入住逻辑只认本系统的标准模型，外部系统经适配器接入；现以 Mock/样例报文跑通，将来对接细节到位仅替换适配器/客户端实现，不动核心逻辑。

**成功标准：**
- 后端 `mvn spring-boot:run` 仍零外部依赖启动（H2 + 种子数据）。
- 可通过 Knife4j 向两个 webhook 端点投递样例报文，生成"待分配"入住意向单。
- 管理员可手工新建意向单；可对意向单选空闲床位并确认入住，生成入住档案，床位/房间占用与统计实时更新，看板/床位页自动反映。
- 重复推送同一业务单据不产生重复入住（幂等）。
- 后端关键逻辑有测试覆盖（沿用 TDD 与现有测试风格）。

## 2. 范围与非目标

**本子项目做：**
- 居住人/员工档案（`dms_resident`）：HCP 同步落地 + 管理员手工维护 + CRUD。
- 入住意向单（`dms_checkin_intake`）：OA/HCP/手工三来源，待分配载体。
- 入住档案（`dms_checkin_record`）：确认入住后生成。
- OA/HCP webhook 接收端点 + 适配器（防腐层）+ 幂等去重。
- 出站 `OaClient/HcpClient` 接口（预留，空实现）。
- 选床确认入住的事务逻辑（占床 + 房间统计更新）。
- 前端「入住管理」菜单组：居住人、入住意向单、入住档案三页。

**非目标（留待后续子项目/阶段）：**
- 退宿、费用结算（后续子项目）。
- OA/HCP 真实对接实现（报文映射细节、真实鉴权、出站回写）——本期仅预留。
- 多节点/部门审批流、员工自助端、电子签名、门禁卡发放、附件上传。
- 智能推荐分配（本期为管理员手工选床 + 校验）。
- 集成事件表/重试/审计（方案 B，本期不做；将来需要时再加，不影响现有领域逻辑）。

## 3. 架构与模块结构

沿用现有"模块化单体 + package-by-feature"。新增三个模块，复用现有 `resource` 模块的床位/房间服务。

```
module/
├── resident/          # 居住人/员工档案
│   └── controller/service/mapper/entity/dto/vo
├── checkin/           # 入住核心：意向单(intake) + 入住档案(record)
│   └── controller/service/mapper/entity/dto/vo
└── integration/       # 对接防腐层
    ├── oa/            # OaCheckinWebhookController + OaApplicationAdapter + 外部 DTO
    ├── hcp/           # HcpEmployeeWebhookController + HcpEmployeeAdapter + 外部 DTO
    └── client/        # OaClient / HcpClient 出站接口（预留，空实现）
```

**模块边界约定：**
- `integration` 只负责"外部报文 → 内部标准命令"，随后调用 `checkin` / `resident` 的 Service 接口，不直接访问 mapper。
- `checkin` 确认入住时，通过 `resource` 模块的 `BedService` / `RoomService` 接口占用床位、更新房间统计，不跨模块直接写表。
- 出站 `OaClient` / `HcpClient` 为预留接口（如"入住完成回写 OA"），现为空实现（log 占位）。

## 4. 数据模型

### 4.4 对现有表的小改
- `dms_bed.current_user_id` 语义明确为"当前居住人 `resident_id`"（原指向 sys_user，DEMO 数据多为空，无迁移负担）。
- 确认入住时：`bed.status` → 已入住、`current_user_id` = resident_id；`room.occupied_beds` +1，必要时 `room.status` → 已满。这些通过 `resource` 模块 Service 完成。

## 5. 对接边界（防腐层）

### 5.1 Webhook 接收端点
```
POST /api/integration/oa/checkin-application   # OA 入住申请单审批通过后推送
POST /api/integration/hcp/employee             # HCP 新员工入职推送
```
- 各自独立的**外部 DTO**（贴外部字段命名），由 `OaApplicationAdapter` / `HcpEmployeeAdapter` 映射成内部 `CreateIntakeCommand`。
- **鉴权（DEMO）**：共享密钥头 `X-Integration-Token`（配置于 application.yml），在 JWT 过滤链之外放行这两个路径并由独立校验拦截；预留真实对接时换成签名校验。
- **幂等**：`biz_no` 唯一约束兜底 + 入库前查重；重复推送返回"已存在/幂等成功"，不报错、不重复建单。
- **HCP 路径**：先按 `employee_no` upsert `dms_resident`，再建 intake。
- **OA 路径**：报文含员工标识 → 按 employee_no 查 resident，无则建（`source` = 3OA）→ 建 intake。
- 原始报文存入 `intake.raw_payload`。

### 5.2 出站预留
- `OaClient.notifyCheckinCompleted(bizNo)`、`HcpClient.*` 等接口先空实现（log 占位），确认入住后调用；将来填真实 HTTP 客户端。

## 6. 入住状态流转

```
[Webhook / 手工] ──创建──> intake(1 待分配)
                                │ 管理员选空闲床位（校验：床位空闲 + 性别符合 gender_limit
                                │                      + 房型符合偏好[不符仅警告]）
                                ▼
                          确认入住 ──> intake(2 已入住) + 生成 checkin_record(1 在住)
                                │        + bed 占用 + room 统计更新（同一事务，任一失败回滚）
                                ▼
                        [退宿子项目] ──> record(2 已退宿) + 释放床位
   intake(1 待分配) ──管理员取消──> intake(3 已取消)
```

**确认入住事务**：生成入住档案 + 占用床位 + 更新房间统计，三步在同一事务内，任一失败整体回滚。

## 7. API 设计

路径前缀 `/api`，统一返回体 `R<T>`，分页 data = `{ records, total, page, size }`。除两个 webhook（走 `X-Integration-Token`）外，其余需 `Authorization: Bearer <token>`。

| 方法 | 路径 | 说明 |
|---|---|---|
| POST | /api/integration/oa/checkin-application | OA webhook（带 token）|
| POST | /api/integration/hcp/employee | HCP webhook（带 token）|
| GET | /api/residents | 居住人分页列表（筛选：姓名/工号/类型/状态）|
| POST/PUT/DELETE | /api/residents(/{id}) | 居住人增/改/删（手工维护）|
| GET | /api/checkin/intakes | 意向单列表（筛选：状态/来源）|
| POST | /api/checkin/intakes | 手工新建意向单 |
| POST | /api/checkin/intakes/{id}/assign | 选床 + 确认入住（核心）|
| POST | /api/checkin/intakes/{id}/cancel | 取消意向单 |
| GET | /api/checkin/records | 入住档案列表/查询（可按楼栋筛选）|

错误处理沿用现有：BizException → 业务错误码 + 提示；参数校验失败 → 400；未认证 → 401；webhook token 错误 → 401。

## 8. 前端设计

新增菜单组「入住管理」，沿用现有 Apple/蓝主题与 ProTable 模式。

- **居住人管理**：列表 + 增改删（与楼栋等页一致的搜索栏 + 表格 + 弹窗）。
- **入住意向单**：列表（按状态 待分配/已入住/已取消 筛选）+「手工新建」+「分配入住」弹窗（联动选 楼栋 → 楼层 → 空闲房间 → 空闲床位，带性别/房型校验提示）+「取消」。
- **入住档案**：只读列表（人 / 床位 / 入住时间），可按楼栋筛选。
- 现有看板页 / 床位页因占用同一套 bed/room 数据，会**自动反映**新入住，无需改动。
