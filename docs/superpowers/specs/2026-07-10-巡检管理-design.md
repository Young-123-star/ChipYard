# 巡检管理首版设计

> 日期：2026-07-10。范围：计划、手工派发、逐项执行、异常整改闭环。

## 目标

在现有楼栋、楼层、房间和数据字典基础上，提供可运行的安全巡检闭环：管理员维护计划，按日期派发任务，巡检人逐项填写结果，异常任务完成整改确认。

## 数据模型

- `dms_inspection_plan`：计划名称、周期、巡检对象、默认巡检人、巡检项 JSON、启停状态。
- `dms_inspection_task`：任务号、计划/对象/巡检项快照、计划日期、执行结果 JSON、整改结果和状态。
- 同一计划和计划日期只有一个任务，由数据库唯一约束保证幂等。
- 任务状态：1 待执行、2 检查通过、3 待整改、4 已整改、5 已取消。
- 对象类型：1 楼栋、2 楼层、3 房间；周期：1 日常、2 每周、3 每月、4 每季度。

计划更新只影响后续任务。已生成任务保存计划名称、对象名称和巡检项快照，历史记录不随资源或计划修改。

## 行为与接口

- `/api/inspection/plans`：分页查询、新建、修改。
- `/api/inspection/plans/{id}/status`：启停计划。
- `/api/inspection/plans/{id}/tasks`：指定日期手工派发；停用计划不能派发。
- `/api/inspection/tasks`、`/tasks/{id}`：分页和详情查询。
- `/api/inspection/tasks/{id}/execute`：提交全部巡检项。异常项必须填写说明；全部正常进入检查通过，否则进入待整改。
- `/api/inspection/tasks/{id}/rectify`：仅待整改任务可确认整改。
- `/api/inspection/tasks/{id}/cancel`：仅待执行任务可取消。

数据库内部使用 JSON 保存快照，接口统一返回结构化的 `items` 与 `results` 数组。巡检项必须来自启用的 `INSPECTION_ITEM` 数据字典。

## 页面

「安全巡检 / 巡检管理」使用单页双标签：计划页负责维护、启停和派发；任务页负责筛选、执行、整改、取消和查看结果。巡检对象复用楼栋→楼层→房间的现有查询接口。

## 非目标

首版不做定时派发、真实用户/RBAC、拍照、定位、电子签名、移动端、公共资产、违规累计、自动转维修工单和导出。需要对应业务或基础模块后再增加。
